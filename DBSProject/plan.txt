🧾 Product Requirements Document (PRD)
📌 Product Name:

Roommate Matching System (Hostel Management)

1. 🎯 Objective

Build a system that helps students find compatible roommates based on lifestyle preferences, habits, and constraints, using structured database design and matching algorithms.

2. ❗ Problem Statement

Students in hostels are randomly assigned roommates, leading to:

Conflicts due to lifestyle differences
Poor study environments
Lack of compatibility (sleep, cleanliness, habits)

Current systems don’t use data-driven matching, only manual allocation.

3. 👥 Target Users
College/hostel students
Hostel administrators
4. ⚙️ Core Features
4.1 User Registration
Student profile creation:
Name, age, course, year
Hostel block / room type
4.2 Preference Collection

Each student inputs structured preferences:

Sleep schedule (early / late)
Study habits (silent / music)
Cleanliness level (low / medium / high)
Smoking / drinking (yes/no)
Guest frequency
Noise tolerance

👉 Store these as attributes in DB (not text blobs)

4.3 Matching Logic

System compares users based on:

Similar preferences
Conflict avoidance (e.g., smoker vs non-smoker)
4.4 Compatibility Score

Each pair gets a score (0–100):

Example logic:

Same habit → +10
Opposite habit → -10
Neutral → +5

Final score determines ranking.

4.5 Roommate Suggestions
Show top 3–5 compatible matches
Allow filtering:
Same course
Same year
Same hostel block
4.6 Admin Allocation
Admin can:
View compatibility scores
Assign rooms manually or auto-assign
5. 🚀 Advanced DBMS Concepts (THIS is where you win marks)
5.1 Many-to-Many Relationships
One student can be compared with many students

👉 Implement using:

Student
Preferences
Compatibility

5.3 Queries You MUST Show

Don’t just say “filtering queries”—show them.

Examples:

Find best matches:
SELECT * FROM Compatibility
WHERE student1_id = ?
ORDER BY score DESC
LIMIT 5;
Find compatible non-smokers:
SELECT s.name
FROM Student s
JOIN Preference p ON s.student_id = p.student_id
WHERE p.smoking = 'no';
5.4 Normalization
Ensure at least 3NF
No redundant data
Preferences stored separately (not inside student)
5.5 Indexing
Index on:
student_id
score (for faster sorting)
6. 🧠 Matching Algorithm (Simple but Defensible)

Don’t overcomplicate. Keep it explainable.

Pseudo logic:

score = 0

if sleep_time same → +10
if study_habit same → +10
if cleanliness same → +10
if smoking conflict → -20
...
